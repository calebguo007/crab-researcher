# CrabRes Agent 架构重新设计——调研报告

> 日期：2026-04-04
> 背景：Agent 端到端测试评分 0-3 分。用户说"更像人机了"。
> 调研对象：Anthropic 官方指南、Claude Code 泄露源码（512K 行）、Manus 架构、Google Agent 设计模式
> 目标：找到根因，设计出真正好用的 Agent

---

## 一、为什么 CrabRes 现在的 Agent 体验是 0 分

### 根因诊断

| 症状 | 根因 | Anthropic 怎么说 |
|------|------|-----------------|
| 连环追问 5-7 轮 | Agent 循环里没有"何时已经够了"的判断 | "代理应在每步从环境获取基本事实来评估进度" |
| 不认识自己是 CrabRes | system prompt 太长，弱模型抓不住关键信息 | "将工具设计视为代理-计算机接口，投入与 HCI 同等的努力" |
| 搜索结果是垃圾但仍开圆桌 | 没有结果质量评估层 | "评估器-优化器模式：一个 LLM 生成，另一个评估" |
| 专家输出泛泛而谈 | 搜索没拿到真数据，专家基于空气分析 | "代理在每步从环境获取基本事实" |
| 输出像模板 | prompt 里写了输出格式，模型机械套模板 | "保持格式接近模型在互联网上自然看到的文本" |

### 最深层的问题

**CrabRes 的 Agent 架构选错了模式。**

我们用的是 **"集群模式"（13 个专家全对全辩论）**——Google 明确说这是"最复杂、成本最高、有无法收敛风险的模式"，只适合"模棱两可的高度复杂问题"。

而用户的需求——"帮我找增长渠道"——更适合 **"协调者模式"或"提示链模式"**：

```
用户输入 → 理解产品(LLM) → 搜索竞品(Tool) → 分析数据(LLM) → 生成策略(LLM)
```

不需要 13 个专家。不需要圆桌辩论。**需要的是一个清晰的流水线，每步做对一件事。**

## 二、Claude Code 是怎么做的（源码分析）

### 核心架构：不是 ReAct Loop，是 Tool-Calling Loop

```
用户输入 → 上下文组装 → API 调用 → 模型返回(文本 or 工具请求)
                                          ↓
                                    [如果是工具请求]
                                    执行工具 → 结果反馈 → 回到 API 调用
                                          ↓
                                    [如果是文本]
                                    返回给用户
```

**关键区别：没有"Coordinator 决定下一步"这个环节。** 模型自己决定是调工具还是直接回答。系统只做两件事：
1. 执行模型请求的工具
2. 检查权限（能不能执行）

### Claude Code 的 5 个关键设计决策

**1. 上下文是第一优先级**

不是"上下文管理"是一个功能，而是**整个架构围绕上下文设计**：
- QueryEngine.ts 有 46000 行——大部分在处理上下文
- 5 种上下文压力策略（时间清除、摘要、记忆提取、历史压缩、截断）
- 动态 token 预算分配

**对比 CrabRes**：我们的 `_build_context` 只有 150 行，上下文管理是事后补丁。

**2. 记忆是提示，不是事实**

> "Agent 在行动前被指示必须根据实际代码库验证其记忆，而不是盲目相信记忆。"

三层记忆：
- Layer 1: MEMORY.md（轻量索引，始终加载，每个指针 ~150 字符）
- Layer 2: 主题文件（按需拉取，不预加载）
- Layer 3: 原始记录（只通过搜索访问）

**对比 CrabRes**：我们把所有记忆（产品+竞品+策略+结果+增长模式）全塞进 context。

**3. 权限和推理分离**

模型生成动作 → 权限系统决定能不能执行 → 不能执行就告诉模型为什么

**对比 CrabRes**：我们的 Trust Levels 是 prompt 里的一行字，不是代码层的拦截。

**4. 子 Agent 隔离**

三种子 Agent 模型：
- Fork（分支）：共享上下文，命中缓存
- Teammate（队友）：独立窗格，文件邮箱通信
- Worktree（工作树）：隔离 git 分支

**关键：子 Agent 只返回输出，不返回完整上下文。** 风险工作不会污染主 Agent。

**对比 CrabRes**：我们的专家直接往 `context["expert_outputs"]` 写，所有专家看到所有人的完整输出。

**5. 后台做梦（Kairos / autoDream）**

用户空闲时，分支子 Agent 在后台运行：
- 合并观察结果
- 消除逻辑矛盾
- 将笔记整合为更清晰的事实

**使用分支子 Agent 做后台维护，防止破坏主 Agent 的工作上下文。**

**对比 CrabRes**：我们的 Growth Dream 也做类似的事，但在午夜才触发，且直接修改主记忆（有破坏风险）。

## 三、Manus 是怎么做的

### 核心架构：Plan → Execute → Verify

```
用户输入 → 任务规划（拆解为步骤列表）→ 逐步执行（每步一个工具）→ 验证结果 → 下一步
```

**Manus 不是对话式的——它是任务式的。** 用户给一个目标，Manus 规划出 10 个步骤，然后一步步执行，每步都有明确的输入输出。

### 关键设计

1. **沙箱环境**：每个任务在 Docker 容器里执行，有真实的浏览器、文件系统
2. **步骤可见性**：用户能看到"步骤 1/10: 搜索竞品 → 步骤 2/10: 分析网站..."
3. **每步独立**：一步失败不影响其他步骤
4. **结果交付物**：最终产出不是"一段话"，而是文件/表格/报告

## 四、Anthropic 官方建议 vs CrabRes 现状

| Anthropic 说 | CrabRes 现状 | 差距 |
|-------------|-------------|------|
| "从最简单的解决方案开始" | 上来就 13 专家圆桌 | 过度设计 |
| "仅在需要时增加复杂性" | 每次对话都走完整 ReAct 循环 | 大部分场景不需要 Agent |
| "工具设计视为 ACI，投入等同 HCI" | 工具定义是简单的 JSON schema | 工具体验差 |
| "格式接近模型自然看到的文本" | 输出格式是 prompt 模板（三阶段结构） | 输出机械化 |
| "在检查点暂停获取人类反馈" | 从头跑到尾不暂停 | 没有人机协作节点 |
| "设置停止条件" | max_iter=10 是唯一停止条件 | 缺少质量停止条件 |

## 五、为什么 Claude（我）有 soul 但 CrabRes 产品没有

这是你问的最好的问题。

**Claude 有 soul 的原因不是模型好——是架构对。**

1. **Claude 不套模板**：没有人在 Claude 的 prompt 里写"第一阶段：市场情报，第二阶段：专家辩论"。Claude 根据问题自然组织回答。CrabRes 强制每次输出都走三阶段模板 → 机械感。

2. **Claude 的回复长度自适应**：简单问题短回答，复杂问题长回答。CrabRes 不管什么问题都想输出完整 Playbook → 压迫感。

3. **Claude 不假装知道**：不确定的就说不确定。CrabRes 搜索到垃圾数据（"CrabRes 定价 $100/mo"、"竞品是 Alteryx"），也硬编成分析 → 不可信。

4. **Claude 记住上下文**：你说过的它不会再问。CrabRes 用户说了 CrabRes 的完整介绍，下一轮还问"能否提供更多信息" → 让人崩溃。

5. **Claude 是一个简单的循环**：输入 → 思考 → 输出/调工具 → 返回结果 → 继续。CrabRes 是 Coordinator → Think → 决定调谁 → 选专家 → 专家分析 → 综合 → 输出 → Critic 审核。**复杂度本身就是 bug 的温床。**

**根本差距：Claude 的架构信任模型的判断力，CrabRes 的架构不信任模型所以用代码控制一切。但代码控制的结果是——模型的 soul 被框架捂死了。**

## 六、重新设计方案

### 原则：简单 > 复杂，流水线 > 圆桌，质量 > 数量

### 新架构：Prompt Chain（提示链）

```
Step 1: UNDERSTAND
  输入: 用户消息 + 历史记忆
  输出: 结构化产品理解 {name, type, audience, goal, budget}
  如果信息不够 → 问 ONE 问题 → 等用户回复 → 重新进入 Step 1

Step 2: RESEARCH  
  输入: 产品理解
  工具: web_search × 3 + social_search × 1（并行）
  输出: 搜索结果摘要
  质量检查: 有效结果 < 2 → 告诉用户"没找到足够数据，你可以提供竞品名或网站帮我精准搜索"

Step 3: ANALYZE
  输入: 产品理解 + 搜索结果
  LLM: 一次调用，不需要 13 个专家
  输出: 竞品分析 + 渠道推荐 + 1 个 hard truth
  
Step 4: PLAN
  输入: 分析结果
  LLM: 一次调用
  输出: 2-3 个 Playbook（但只有用户选择的那个才展开细节）
  
Step 5: EXECUTE (只在用户说"开始执行"后)
  此时才需要专家——copywriter 写文案、designer 做图
  每次只调 1 个专家做 1 件具体的事
```

### 与现在的对比

| 维度 | 现在 | 新方案 |
|------|------|--------|
| 架构 | ReAct Loop + 圆桌集群 | Prompt Chain（流水线） |
| LLM 调用次数 | 10-20 次（coordinator × N + 专家 × 4 + 综合） | 4-5 次 |
| 专家使用 | 每次都调 4 个专家圆桌 | 只在执行阶段按需调 1 个 |
| 用户感受 | "它在忙，但我不知道在干嘛" | "它在做第 2/4 步：搜索竞品..." |
| 输出格式 | 强制三阶段模板 | 自然语言，像 Claude 一样 |
| 成本 | ~100K tokens / 次 | ~20K tokens / 次 |
| 耗时 | 3-5 分钟 | 30-60 秒 |

### 关键改动清单

1. **砍掉 Coordinator 决策层**
   - 不再让 LLM 决定"下一步做什么"
   - 用代码写死流水线：理解 → 搜索 → 分析 → 计划
   - 模型只负责每一步的内容生成

2. **砍掉默认圆桌**
   - 普通对话不开圆桌
   - 只在用户明确要求"帮我执行"时才调专家
   - 每次只调 1 个专家做 1 件事

3. **输出去模板化**
   - 删掉 Coordinator prompt 里的"三阶段结构"
   - 让模型自然回答
   - 只保留 hard truth 这一个约束

4. **步骤可视化**
   - 前端展示"步骤 1/4: 理解你的产品 ✓"
   - 不再显示"Coordinator deciding next tactical move..."
   - 用户能看到进度，不焦虑

5. **上下文重新设计**
   - 每步只给该步需要的 context（不是全量）
   - Step 1 只需要用户消息 + 历史记忆
   - Step 2 只需要产品理解 + 工具结果
   - Step 3 只需要产品理解 + 搜索摘要

## 七、实施优先级

### Phase 1：流水线替代 ReAct Loop（最关键）
- 写一个新的 `PipelineRunner` 替代 `AgentLoop`
- 4 步流水线，每步是一个独立的 async 函数
- 不用 function calling，直接在代码里调工具

### Phase 2：输出自然化
- 删掉三阶段模板
- 让模型根据搜索数据自然回答
- 只保留 2 个硬约束：(1) 引用具体数据 (2) 一个 hard truth

### Phase 3：步骤可视化
- 前端 Chat 页展示步骤进度条
- 每步完成后自动展示该步的摘要结果

### Phase 4：按需专家
- 用户选择 Playbook 后才调专家
- copywriter 写文案、designer 出图
- 每次 1 个专家做 1 件事
