## 一、为什么 OpenClaw + Skills 不是终解

通过深度调研 OpenClaw、Claude Code 源码泄露、Harness Engineering，以及三者的架构对比，结论很清晰：

### OpenClaw 的 5 个根本问题

| # | 问题 | 本质 |
|---|------|------|
| 1 | **单 Agent 架构** | OpenClaw 本质是一个 Gateway + Brain + Hands 的单体。虽然 v2026.3.24 加了 OpenWebUI 子 Agent 编排，但这是后补的，不是原生设计。它像一个全能选手，而非一个团队。 |
| 2 | **Skills 是拼凑的，不是编排的** | 每个 Skill 是独立的 Markdown Prompt 文件，执行完就结束。Skill A 的输出不会自动流入 Skill B。用户得自己当粘合剂。 |
| 3 | **无长程规划能力** | 增长策略需要多轮研究→分析→假设→验证→调整。OpenClaw 的对话模型是一轮一答，缺乏跨多轮的规划记忆和阶段管理。 |
| 4 | **无主动性** | OpenClaw 是被动的——你问它才答。它不会主动说「竞品刚涨价了」「你该发帖了」「上周策略效果不好该调了」。 |
| 5 | **无领域深度** | Skills 是通用的。`marketing-ideas` 给你 139 条通用建议，但不会基于你的产品、你的预算、你的竞品来定制。缺乏 Context Engineering。 |

### Claude Code 做对了什么（CrabRes 要学）

| Claude Code 的设计 | 为什么这很重要 | CrabRes 怎么用 |
|-------------------|--------------|---------------|
| **ReAct Loop** — think→act→observe 循环多轮 | 增长研究天然是多轮的：搜→分析→发现新问题→再搜 | Agent Loop 核心 |
| **子 Agent 专业化** — Planner/Critic/Explorer 各司其职 | 增长涉及 12 个专业领域，一个大脑不够 | 专家圆桌会议 |
| **流式工具执行 + 并发** — 不等完整响应就开始执行 | 研究阶段需要同时抓多个竞品、多个平台 | 并发研究 |
| **memdir/ 4 类记忆** — user/feedback/project/reference | 增长是持续过程，需要记住产品、策略、效果 | 记忆层 |
| **上下文预算管理** — 动态分配 Token 给不同模块 | 避免一个专家用完所有 Token | 专家资源调度 |
| **权限 5 道防线** — Prompt→Schema→审批→工具验证→Hook | 防止 Agent 乱花钱、乱发帖、乱联系人 | Harness 控制层 |
| **会话检查点/恢复** — 完整状态快照 | 增长研究可能跨天进行 | 长程状态管理 |
| **CLAUDE.md 宪法** — 项目级行为规范 | 用户的品牌调性、预算约束需要全局生效 | growth-config |

### Harness Engineering 怎么融进来

Harness 的五层正好对应 CrabRes 的控制需求：

| Harness 层 | CrabRes 对应 | 具体设计 |
|-----------|-------------|---------|
| **工具层** | 专家的双手 | 专用工具 > 通用工具。`search_reddit` 而非 `bash curl` |
| **上下文层** | 用户的产品 DNA | `growth-config`：产品信息/目标/约束/品牌调性，全局注入所有专家 |
| **执行层** | 圆桌会议编排 | 首席增长官调度专家，按阶段推进，不是所有人同时说话 |
| **权限层** | 分级执行 | 研究=自动，内容生成=自动，发送=确认，花钱=禁止 |
| **记忆层** | 越用越准 | 记住什么策略有效、什么竞品已分析、用户偏好什么 |

## 二、CrabRes 的核心竞争力：圆桌会议

这是 CrabRes 和 OpenClaw + Skills 的根本区别。

### OpenClaw 模式（拼凑式）
```
用户 → "帮我做增长计划"
  ↓
OpenClaw 调用 competitor-analysis Skill → 输出竞品分析
  ↓ (手动)
用户 → "基于这个结果帮我做 Reddit 策略"
  ↓
OpenClaw 调用 reddit-marketing Skill → 输出 Reddit 建议
  ↓ (手动)
用户 → "帮我写帖子"
  ↓
OpenClaw 调用 social-post-writer Skill → 输出帖子

问题：
· 每个 Skill 独立运行，互不知道对方的结果
· 用户要自己当粘合剂和协调者
· 没有人检查策略一致性
· 没有人关心预算够不够
· 没有人追问"两周后效果如何"
```

### CrabRes 模式（圆桌会议式）
```
用户 → "帮我做增长计划"
  ↓
首席增长官（Coordinator）：
  "好。我先让市场研究和经济学顾问做前期调研。"
  ↓
┌─── 圆桌第 1 轮：研究 ────────────────────────────┐
│                                                   │
│  市场研究专家：                                     │
│    "我查了 5 个竞品，发现竞品 A 流量主要来自         │
│     Reddit (42%) 和 SEO (35%)。竞品 B 主要靠        │
│     YouTube 博主合作。"                             │
│                                                   │
│  经济学顾问：                                       │
│    "基于用户预算 $200/月，YouTube 博主合作平均        │
│     $500/次，不现实。Reddit 免费获客 + SEO 长期       │
│     投入是最优解。广告暂不推荐。"                     │
│                                                   │
│  首席增长官：                                       │
│    "同意。但我注意到竞品 A 的 Reddit 策略主要是       │
│     在 r/resumes 发帖。让我问问用户的产品是否         │
│     适合这个社区。"                                  │
│                                                   │
│  → 向用户提问：                                     │
│    "你的产品主要面向应届生还是资深职场人？             │
│     这决定了我们选择哪些 Reddit 社区。"               │
└───────────────────────────────────────────────────┘
  ↓ (用户回答后)
┌─── 圆桌第 2 轮：策略 ────────────────────────────┐
│                                                   │
│  社媒运营专家：                                     │
│    "根据用户面向应届生的定位，推荐 3 个社区：         │
│     r/cscareerquestions, r/resumes, r/jobs。        │
│     每个社区的调性不同，我设计了差异化的内容策略。"    │
│                                                   │
│  消费心理学专家：                                    │
│    "应届生群体的核心焦虑是'不确定性'。               │
│     内容应该用'降低风险'的框架，而不是'提升能力'。     │
│     比如'避免简历里的5个致命错误'比'5个优化简历        │
│     的技巧'转化率高 40%。"                           │
│                                                   │
│  经济学顾问：                                       │
│    "按这个策略，预计月 CAC 为 $0.5-1.2（纯时间        │
│     成本）。如果每天投入 30 分钟，月可触达 2000       │
│     目标用户，转化率按 2% 估算约 40 注册/月。         │
│     3 个月达到 1000 用户需要同时启动 SEO 长线。"      │
│                                                   │
│  首席增长官：                                       │
│    "策略明确了。让文案大师开始写内容，                 │
│     内容营销专家做 SEO 关键词研究。"                  │
└───────────────────────────────────────────────────┘
  ↓
┌─── 圆桌第 3 轮：执行物料 ────────────────────────┐
│                                                   │
│  文案大师 + 内容营销专家并行工作                     │
│  产品增长专家补充产品侧建议                         │
│  数据分析师设计 KPI 追踪表                          │
│                                                   │
│  消费心理学专家审核所有文案的说服力                   │
│  经济学顾问审核整体预算合理性                        │
│                                                   │
│  首席增长官汇总为最终交付物                          │
└───────────────────────────────────────────────────┘
  ↓
输出：完整增长计划 + 执行物料 + KPI 表

  ↓ (2 周后)
首席增长官主动发起：
  "两周复盘时间到了。给用户留任务：
   请提供这两周的执行数据。"
  ↓
┌─── 圆桌第 4 轮：复盘迭代 ────────────────────────┐
│  根据实际数据调整策略...                            │
└───────────────────────────────────────────────────┘
```

### 区别的本质

| 维度 | OpenClaw + Skills | CrabRes |
|------|------------------|---------|
| 知识 | 每个 Skill 有独立知识 | 所有专家共享同一个上下文（用户产品+竞品数据+预算） |
| 协作 | 无（串行调用，互不知道） | 圆桌会议（专家之间互相参考、质疑、补充） |
| 一致性 | 无人检查 | 首席增长官 + Critic 确保策略不矛盾 |
| 深度 | 每个 Skill 独立判断 | 多维度交叉验证（心理学+经济学+数据验证同一个策略） |
| 连续性 | 每次调用是独立的 | 跨会话记忆，知道上次策略效果如何 |
| 主动性 | 被动等待 | 主动追踪、主动复盘、主动发现机会 |

## 三、用户看到什么

你说得对——不能让 12 个角色的任务/输出铺满屏幕让用户眼花缭乱。

### 设计原则：**圆桌在后台，用户看到的是一个声音**

```
用户视角：
┌──────────────────────────────────────────────────┐
│                                                  │
│  CrabRes                                         │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  💬 对话区（主界面）                        │  │
│  │                                            │  │
│  │  CrabRes:                                  │  │
│  │  "我分析了你的产品和 5 个竞品，             │  │
│  │   发现 Reddit 和 SEO 是最优渠道。            │  │
│  │   预算方面，$200/月足够启动这两个策略。      │  │
│  │                                            │  │
│  │   你的产品主要面向应届生还是资深职场人？     │  │
│  │   这会影响我选择哪些社区和关键词。"          │  │
│  │                                            │  │
│  │  [展开研究过程 ↓]  ← 可选查看              │  │
│  │                                            │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  📋 待你完成（统一收口，不分角色）          │  │
│  │                                            │  │
│  │  □ 回答上面的问题                          │  │
│  │  □ 提供 Google Analytics 权限（可选）       │  │
│  │                                            │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
└──────────────────────────────────────────────────┘
```

**关键**：
- 对话中 CrabRes 是**一个声音**，不是 12 个角色分别发言
- 首席增长官负责汇总所有专家意见，用统一口吻输出
- 待办任务**不分角色**，统一呈现，按紧急度排序
- 用户可以点「展开研究过程」查看圆桌讨论——这是**可选的透明度**

### 展开后看到什么

```
[展开研究过程 ↓]
┌──────────────────────────────────────────────┐
│  🔍 市场研究专家                              │
│  搜索了 5 个竞品的流量来源...                  │
│  发现: 竞品 A - Reddit 42%, SEO 35%           │
│  发现: 竞品 B - YouTube 博主合作为主            │
│                                              │
│  💰 经济学顾问                                │
│  $200/月预算分析:                              │
│  · YouTube 博主 $500/次 → 超预算              │
│  · Reddit 免费 + SEO 长线 → 最优解             │
│                                              │
│  🧠 消费心理学专家                             │
│  应届生焦虑分析:                               │
│  · "降低风险" > "提升能力" 转化率高 40%         │
│                                              │
│  → 首席增长官综合以上得出结论                   │
└──────────────────────────────────────────────┘
```

**这像什么？** 像你和一个高级顾问开会。他一个人给你讲，但你知道他背后有一个团队做了详细研究。你想看细节可以看，不想看就信任他的结论。

## 四、统一的任务系统

你说的对——不能让用户面对 12 个角色的任务。设计如下：

### Agent → 用户的任务（统一队列）

```
任务产生过程（后台）：
  市场研究专家想知道 GA 数据 → 创建任务
  社媒专家想知道品牌调性 → 创建任务
  经济学顾问想知道预算 → 创建任务
       ↓
  首席增长官 审核+合并+排序：
  · 合并重复的（多个专家都想知道预算 → 合并为一个）
  · 排优先级（阻塞后续研究的排前面）
  · 必要的才留，可选的标为可选
       ↓
  用户看到的：
  ┌──────────────────────────────────┐
  │  📋 待你完成                     │
  │                                  │
  │  1. 你的月度营销预算是多少？      │
  │     [填写]                       │
  │                                  │
  │  2. 产品面向应届生还是资深人士？   │
  │     [应届生] [资深] [都有]        │
  │                                  │
  │  3. (可选) 接入 Google Analytics  │
  │     [授权] [跳过]                │
  └──────────────────────────────────┘
```

**原则**：
- 用户永远看到一个统一的待办列表
- 不超过 3-5 个任务同时展示
- 标注哪些是必须的、哪些是可选的
- 每个任务解释**为什么需要这个信息**（一句话）

## 六、和 OpenClaw + Skills 的根本差距

| 维度 | OpenClaw + Skills | CrabRes |
|------|------------------|---------|
| **架构** | 单 Agent + 插件拼凑 | 圆桌会议 + 编排器 + 统一输出 |
| **知识共享** | 每个 Skill 独立上下文 | 所有专家共享完整记忆 |
| **质量控制** | 无 | Critic 审核 + 心理学审核 + 经济学审核 |
| **用户体验** | 用户当粘合剂 | 一个声音、一个待办列表 |
| **主动性** | 被动等待 | 后台引擎持续工作 |
| **连续性** | 每次对话独立 | 跨会话跨周跨月的增长伙伴 |
| **深度** | 通用建议 | 基于你的产品/预算/竞品定制 |

**一句话：OpenClaw 给你一盒工具，CrabRes 给你一个团队。**
